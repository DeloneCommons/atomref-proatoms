# Multiwfn interoperability data

This README is not generated by the export script; it is a committed data-contract file.

This directory stores Multiwfn-oriented interoperability files generated from the
same local SCF checkpoint, NPZ, and metadata artifacts that support the released
atomref-proatoms profile layer. It is a data contract page: it describes the
intended file layout, basis-format policy, validation checks, and regeneration
commands. For the scientific interpretation of proatoms as reference gauges, see
`docs/theory.md`, `docs/methods.md`, and `docs/discussion.md`.

Multiwfn files are derived interoperability products. They do not replace the
project-native profile, radii, QA, comparison, or structured local SCF artifacts.
The preferred internal data path remains the package-native SCF/profile/metadata
layer. Multiwfn files are provided for workflows that expect Multiwfn-style
atomic radial densities or PROAIM WFN inputs.

## Current export policy

The active profile configuration declares the following Multiwfn outputs:

```text
pbe0_sfx2c_x2cqzvpall_h-rn_spherical_v2
  basis: x2c-QZVPall
  .rad: all H-Rn configured states
  .wfn: neutral H-Rn atoms

pbe0_sfx2c_dyallv4z_h-lr_spherical_v2
  basis: dyall-v4z
  .rad: all H-Lr configured states
  .wfn: not emitted as a dataset-wide product

pbe0_sfx2c_x2cqzvpalls_h-rn_spherical_v2
  basis: x2c-QZVPall-s
  .rad/.wfn: not emitted; basis-sensitivity branch

pbe0_sfx2c_dyallav4z_h-ba_hf-ra_spherical_v2
  basis: dyall-av4z
  .rad/.wfn: not emitted; basis-sensitivity branch
```

The `.rad` files are density-only products and are the compact export path for
stockholder/Hirshfeld-like workflows. They are generated by evaluating the local
SCF density on the fixed Multiwfn `atmrad` radial grid. The default export uses
a fixed Cartesian ray of the one-center spherical SCF density, while an
angular-average mode is available for local diagnostics. They are not generated
by interpolating the committed `profiles.csv` tables.

The `.wfn` files are wavefunction-like interoperability containers. They also
require local SCF checkpoint and NPZ arrays because the profile CSVs do not
contain orbital coefficients or primitive Gaussian information. The current WFN
policy is conservative: neutral `x2c-QZVPall` atoms are configured because that
branch does not exceed g primitives, while `dyall-v4z` is not emitted as a
dataset-wide WFN product. The current Multiwfn manual and source support WFN
h-type primitive labels, but Multiwfn's Gaussian-function support stops at h and
the `dyall-v4z` bundle contains i-shell primitives for part of the f-block. A
partial dyall WFN subset would therefore have a different coverage contract from
the dyall radial profile branch and is deferred until a separate basis/format
policy is chosen.

The manifest is committed as `data/multiwfn_artifacts/manifest.json`. It records
profile-data version, dataset configuration path, generated file list, per-format
counts, local SCF source paths, and per-file validation metadata. Paths in the
manifest are repository-relative. The optional `file` field is retained only as a
compatibility alias for older diagnostics and must match the canonical `path`
field.

## Intended Multiwfn use

The `.rad` and `.wfn` files make the atomref-proatoms reference gauge available
to Multiwfn analyses that use atomic densities or atomwfn-style inputs. Examples
include promolecular and deformation-density maps; Hirshfeld/Hirshfeld-I-like
charge and population analysis; VDD and ADCH-style charge analysis; fuzzy
atomic-space integration; orbital composition based on Hirshfeld partitioning;
real-space density, spin-density, gradient, Laplacian, ELF/LOL, ALIE, and source
function maps; topology, basin, domain, and molecular-surface analyses; weak
interaction visualizations such as RDG/NCI, IGM/IGMH/mIGM/aIGM, IRI, and DORI;
and density-difference or charge-transfer inspection.

The main scientific advantage is that atomref-proatoms supplies self-consistent
spherical reference densities. The open-shell sphericalization is built into the
SCF model by distributing occupations over angular-momentum manifolds; it is not
just an angular average applied after an ordinary anisotropic open-shell atom has
converged. The product also gives a broader versioned coverage contract than a
small ad hoc atomwfn folder: primary-branch `.rad` coverage spans all configured
H--Rn and H--Lr states, and the neutral H--Rn `.wfn` subset follows a documented
basis-format boundary.

## Layout

Generated files are written under:

```text
data/multiwfn_artifacts/
  README.md
  manifest.json
  rad/
    <dataset_id>/
      H_0.rad
      O-1.rad
      Fe+2.rad
      ...
  wfn/
    <dataset_id>/
      H .wfn
      He.wfn
      ...
```

`.rad` filenames follow Multiwfn's charge-suffix convention: neutral atoms use
`_0`, cations use `+q`, and anions use `-q`. One-letter neutral WFN atom
filenames intentionally retain Multiwfn's space-padded atomwfn convention, for
example `H .wfn` and `O .wfn`. This space is part of the intended compatibility
surface, not a typo.

A manifest produced by a format-restricted run documents the files generated in
that run. A complete release manifest should be regenerated after all intended
`.rad` and `.wfn` files have been written.

## Validation checks

The checker re-reads every generated `.rad` file, verifies the fixed Multiwfn
`atmrad` grid, and checks that the finite-grid integral, SCF-source metadata,
and evaluation mode are consistent with the manifest. It deliberately does not
compare `.rad` values to interpolated profile tables, because the release `.rad`
product is required to be SCF-derived. For `.wfn` files, the checker uses the
package validation parser to verify electron-count semantics. These are
format-boundary checks: they do not implement a Hirshfeld-I iteration, and they
do not make Multiwfn a Python dependency.

A separate H/O/H2O documentation notebook validates the PROAIM WFN convention
against PySCF and Multiwfn for representative total-density, spin-density, and
deformation-density use. That validation supports the interoperability path but
is not a universal proof for every element, charge, basis, angular momentum, or
future Multiwfn build.

## Regeneration

The `.rad` and `.wfn` exports both require local SCF artifacts under
`local-data/scf/`:

```bash
python scripts/export_multiwfn_artifacts.py --dry-run
python scripts/export_multiwfn_artifacts.py --format rad --force --check
python scripts/check_multiwfn_artifacts.py --require-generated
```

The default `.rad` evaluation uses one fixed ray of the spherical SCF density;
for a slower local angular-average diagnostic, add `--rad-angular-points 110`.

To generate a complete manifest after both density-only and WFN products are
available, run the combined export explicitly:

```bash
python scripts/export_multiwfn_artifacts.py --format all --force --check
python scripts/check_multiwfn_artifacts.py --require-generated
```

For WFN-only local debugging, a format-restricted run is also available; this
rewrites the manifest for that run and should be followed by the combined export
before committing a complete product manifest:

```bash
python scripts/export_multiwfn_artifacts.py --format wfn --force --check
```

For inspection without writing files:

```bash
python scripts/export_multiwfn_artifacts.py --dry-run
python scripts/export_multiwfn_artifacts.py --list --show-jobs
```
